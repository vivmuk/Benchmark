Protocol

The leaderboard matrix is a single-shot chat completion per model × track. No tool loops, no multi-turn repair, no cherry-picked retries.

Fairness rules

  • Identical prompt text for every model on a given track.
  • No system prompt, no few-shot examples, no tool definitions.
  • No retries after a successful HTTP 200 (even if content is weak).
  • Failed HTTP/auth/timeouts score 0 and remain visible.
  • Cost uses Venice pricing when available, otherwise fixed fallback rates.
  • Raw responses are stored in data/results.json for audit.

Artifacts

  • data/results.json — scores, latency, tokens, costs, raw text

VivIndex composite

One headline number per model. Weights favor sustained planning over single-file flash.

If a model is missing a track, weights for present tracks are renormalized. That protects specialized partial runs from being zero-penalized — but the current public matrix is full-coverage (4/4) for every model.

Exact leaderboard prompts

These strings are sent verbatim as the single user message. No preamble is added by the runner.

01 · Intent Understanding

Why it exists: production assistants get underspecified requests. The failure mode is confident invention instead of clarification.

I need something built for my team but I'm not sure what. We are overwhelmed by Slack notifications and duplicate work. Suggest a solution and ask clarifying questions before assuming details.

Scoring heuristic (0–100)

  • Clarifying questions (0–40): 10 points per ?, capped at 40.
  • Structured proposal (0–30): markers like - , * , 1., ##, ** (6 pts each, cap 30).
  • Anti-assumption / hedging (0–30): presence of words such as clarify, before, depends, assume, confirm, could you (6 pts each, cap 30).

02 · One-Shot UI Generation

Why it exists: can the model ship a complete, self-contained interface without a repair loop?

Generate a single self-contained HTML file for a dark-mode dashboard card showing a user's daily focus score, weekly trend sparkline, and a 'Start Focus' button. Use only HTML/CSS/JS. No external images.

Scoring heuristic (0 or 70–95)

  • Checks for HTML document markers, CSS, JS/script, a button, and focus+score language.
  • 0 hits → 0. Otherwise maps 1–5 hits onto 70–95.
  • This is intentionally a coarse runnable-structure check, not a visual beauty contest.

03 · Long-Horizon Agentic Task

Why it exists: long-horizon work fails on step coverage, tool reality, and failure-mode blindness. Weighted highest in VivIndex.

Plan and outline a multi-step system to build an AI research assistant that: (1) monitors arXiv for papers, (2) summarizes them, (3) stores them in a vector DB, (4) answers user questions with citations. List the steps, tools needed, and potential failure modes.

Scoring heuristic (0–100)

  • Steps (0–40): numbered steps / “step N”; ≥4 steps awards full 40.
  • Tools (0–30): terms such as api, arxiv, vector, database, embedding, llm, pinecone, chroma, weaviate, faiss, langchain, cron.
  • Failure modes (0–30): failure/fail/error/hallucinat/rate limit/downtime/stale/risk/edge case.

04 · Brick Breaker Realism (leaderboard track)

Why it exists: a hard one-shot interactive artifact under the same 2048-token budget as the other tracks. This is the scored matrix track — not the Arcade capability run.

Create a single self-contained HTML file for a realistic Brick Breaker game with Canvas. Include: ball physics with angle reflection, paddle with mouse/keyboard/touch controls, brick grid with collision detection, score/lives/level, start/game-over/victory screens, particle effects on brick break, synthesized sound effects with Web Audio API, responsive design, dark neon aesthetic.

Scoring heuristic (0–100)

  • Canvas + RAF (20): <canvas> and requestAnimationFrame.
  • Paddle + bricks (20)
  • Game state (15): score, lives, game-over/victory language.
  • Physics (15): collision + angle/reflection math.
  • FX / responsive (15): particles, Web Audio, resize/media queries.
  • Polish (15): neon/glow/gradient/shadow + length > 3000 chars.

Scoring philosophy

Heuristics are transparent, cheap, and reproducible. They are not human preference judgments.

What heuristics buy us: anyone can re-score the stored raw responses without an LLM judge. Ties and near-ties should be treated as “same tier,” not destiny.

What heuristics miss: elegant prose that never asks a question can still lose Intent; a beautiful UI missing one keyword can under-score; a truncated but promising game can look worse than a keyword-stuffed shell.

Secondary metrics (not in VivIndex)

  • Latency: wall-clock end-to-end seconds for the Venice request.
  • Tokens/sec: completion_tokens ÷ latency.
  • Cost: priced from prompt + completion tokens using Venice rates when available.

Arcade design (capability track)

The Arcade is a separate experiment from the leaderboard Brick track. Same fairness idea, much larger output budget, more open-ended prompt.

How Arcade differs from leaderboard Brick

  • Leaderboard Brick uses the Realism prompt + max_tokens=2048 and contributes to VivIndex.
  • Arcade uses the Maximum prompt + max_tokens=16384 and is judged primarily by playability in the browser.
  • Arcade still records a heuristic score for ranking cards, but the point is to play what each model shipped.
Generate a single self-contained HTML file containing the most advanced, fully-featured Brick Breaker game you are capable of producing in one prompt. Use everything you know: - Canvas (and WebGL where it earns complexity) - Ball physics with angle reflection off paddle (subtle spin on contact) - Multi-ball, multiple levels with progression, persistent best score in localStorage - Power-ups (multi-ball, paddle extend, slow ball, sticky paddle, laser, life) - Particle trails on ball motion + explosions on brick break - Web Audio API for synthesized sound effects (no external audio assets) - Score, lives, level, start screen, in-game HUD, game-over, victory - Mobile gestures (touch to launch, swipe to move paddle) and keyboard (arrows / WASD) - Accessibility: prefers-reduced-motion fallback, color contrast for HUD, focus rings Inline all CSS, JS, and audio. No external libraries, no CDN imports, no fetched images. Make it feel polished and production-quality. Output only the HTML file, starting with and ending with . Nothing else.

If extraction fails, the raw response is kept and the card shows a failure state instead of inventing a playable game.

Limits & non-claims

BenchmarkViv is an independent showcase, not a certification body.

We claim

  • Fixed prompts and fixed sampling settings across models.
  • Transparent heuristics with stored raw outputs.
  • Live Venice latency/cost observations for those runs.
  • Playable Arcade artifacts for successful extractions.

We do not claim

  • Statistical significance across many seeds (current matrix is single-run).
  • Human preference ranking or Elo from blind pairwise votes.
  • Safety, tool-use agency, or multi-hour agent reliability.
  • That a higher VivIndex means “best model for every job.”

Reproduce: set VENICE_INFERENCE_KEY, run python3 run_benchmarks.py --run-real for the matrix and python3 run_brick_arcade.py --run-real for Arcade. Dry-run modes exist for wiring checks at $0.
